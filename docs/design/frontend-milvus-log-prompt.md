# Frontend Milvus Log Rendering Prompt

## Objective

Implement the frontend experience for Milvus log viewing based on structured log data returned by Java.

The frontend must focus on rendering, styling, filtering, searching, and streaming interaction. It must not parse the raw protocol prefixes emitted by the agent.

## Upstream Contract

The Java service is responsible for reading the agent container logs and parsing the stable forwarded-log protocol prefixes:

- `[MILVUS-STDOUT] `
- `[MILVUS-STDERR] `

The frontend should consume structured log entries instead of parsing raw lines.

## Expected Input Model

Assume the frontend receives log entries shaped like this:

```json
{
  "sequence": 1,
  "stream": "stdout",
  "parsed": true,
  "timestamp": "2026-03-24T10:00:00Z",
  "level": "INFO",
  "message": "start query node",
  "rawMessage": "2026-03-24 18:00:00 INFO start query node",
  "rawLine": "[MILVUS-STDOUT] 2026-03-24 18:00:00 INFO start query node"
}
```

Field meanings:

- `sequence`: stable ordering field generated by Java
- `stream`: `stdout`, `stderr`, or `unknown`
- `parsed`: whether the message was successfully structured by Java
- `timestamp`: parsed timestamp when available
- `level`: normalized level such as `INFO`, `WARN`, `ERROR`
- `message`: primary display content
- `rawMessage`: original content after protocol prefix removal
- `rawLine`: original full line for fallback display

## Frontend Responsibilities

The frontend must:

1. render historical and streaming log entries
2. visually differentiate stdout and stderr
3. visually differentiate log levels
4. support searching and filtering
5. support auto-scroll and pause auto-scroll
6. gracefully handle missing or partially parsed fields

The frontend must not:

1. parse `[MILVUS-STDOUT] ` or `[MILVUS-STDERR] ` directly
2. infer stdout/stderr by string matching raw lines
3. rely on unstable raw text patterns for core behavior

## Rendering Rules

### Main row content

Each log row should display, when available:

1. timestamp
2. stream badge
3. level badge
4. message text

Fallback order:

1. `message`
2. `rawMessage`
3. `rawLine`

### Stream styling

Recommended styling:

- `stdout`: neutral or normal style
- `stderr`: warning or stronger style
- `unknown`: muted fallback style

### Level styling

Recommended styling:

- `TRACE`: faint
- `DEBUG`: subdued
- `INFO`: normal
- `WARN`: yellow/orange
- `ERROR`: red
- `FATAL`: strong red
- `UNKNOWN`: gray

## Required Interactions

Implement at least:

1. show all logs
2. show only stdout
3. show only stderr
4. keyword search
5. clear local buffer
6. auto-scroll to newest logs
7. pause auto-scroll
8. streaming connection state indicator

## Data Flow Recommendation

Recommended flow:

1. initial request loads recent history
2. a streaming channel appends new logs in real time
3. ordering is stabilized by `sequence`
4. duplicate insertion is avoided when reconnecting

Recommended transport options:

1. SSE preferred
2. WebSocket acceptable if already standardized in the frontend stack

## Performance Requirements

Log UIs can become large quickly. Design for:

1. virtualized or incremental rendering
2. bounded in-memory retention, for example 1000 to 2000 lines
3. efficient filtering without full re-render on every message
4. stable append performance during streaming

## Empty and Error States

Provide clear UI states for:

1. no logs yet
2. loading history
3. streaming connected
4. streaming reconnecting
5. streaming disconnected
6. backend error

## Accessibility and Usability

Please keep the log UI readable:

1. use monospace font for log content
2. preserve whitespace and line alignment
3. make badges compact but clear
4. keep colors readable in both light and dark themes if applicable

## Deliverables

Please provide:

1. page or component structure
2. state design
3. rendering strategy
4. filtering/search UX
5. SSE or WebSocket integration design
6. style rules for stream and level badges
7. fallback handling for incomplete log entries
8. a minimal implementation-first frontend solution
